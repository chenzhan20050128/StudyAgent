# StudyAgent 技术方案文档 

更新时间：2026年4月2日

关键约束与总体原则

1 可运行优先
   后端以 FastAPI 为入口，业务逻辑沉在 Service 与 Agent 中，测试脚本用于端到端验证

2 确定性工作流优先
   计划生成与测验生成都采用确定性步骤编排，LLM 只参与可解释的节点，例如扩展目标 生成大纲 生成题目 主观题评分

3 用户数据隔离
   所有 MySQL 与 Milvus 数据都按 user_id 做逻辑隔离

4 真实资料优先
   无论计划还是测验，尽量要求 LLM 基于用户知识库真实片段输出，不鼓励凭空编造

5 失败可降级
   当 langgraph 或某些依赖不可用时，工作流走顺序执行兜底
   当结构化解析失败时，返回可接受的降级结果而非中断

   FastAPI 应用入口，组织路由与依赖实例化

quiz_service.py

chat_agent.py

quiz_chat_agent.py

main_chat_agent.py

三 关键技术栈与外部依赖

后端

Python 3.12
FastAPI 作为 HTTP API 框架
SQLAlchemy 作为 ORM
MySQL 作为业务数据存储

向量检索

Milvus 作为向量数据库
稠密向量用于语义检索
稀疏向量或 BM25 用于关键词补充
项目使用混合检索并做加权融合

大模型与相关模型

LLM 通过 OpenAI compatible 接口访问，默认对接 DashScope
embedding 模型用于向量化
rerank 模型用于重排
chat 模型用于生成回答 生成大纲 生成题目 主观题评分 结构化意图抽取

解析与清洗

PDF 使用 PyMuPDF
Word 使用 python docx
网页使用 requests 与 BeautifulSoup


四 数据模型与存储设计

1 MySQL 负责权威业务状态
   文档元信息 标签 学习计划 每日任务 测验题目 答题记录 薄弱点 复习排期 RAG 查询日志

2 Milvus 负责知识片段的高效检索
   每条记录对应一个 chunk
   关键字段包含 user_id doc_id chunk_id content embedding 以及必要的元信息

3 数据隔离
   MySQL 查询全部带 user_id 条件
   Milvus 检索 filter_expr 带 user_id 条件，可选追加 doc_id in 列表

4 不把 chunk 明细落 MySQL
   目的是降低关系库压力，避免大量 chunk 行导致复杂查询与迁移成本
   chunk 的可解释性通过 source_chunks 或查询日志持久化关键引用即可


五 文档入库与知识库构建

需求对应

学习资料管理
支持多格式上传与 URL
自动解析提取内容 构建个人知识库
支持标签分类

总体流程

1 接收上传
   接口层接收文件或 URL
   创建 documents 记录并保存文件到 data 目录

2 解析为纯文本
   parsers.py 负责按类型解析
   输出 raw_text 与适当的元信息
   解析失败时标记文档状态并记录原因

3 语义分块
   真正分块由 LangChain 的 RecursiveCharacterTextSplitter 完成
   分块参数为 chunk_size 800 overlap 200
   parsers.py 的 DocumentParser 明确只解析不分块

4 生成 embedding
   llm_client.py 提供 embed 能力
   设计了 embedding_cache.json 落盘缓存，按文本 sha256 去重
   embedding 按批处理降低请求次数

5 写入 Milvus
   vector_store.py 负责建表与索引
   文档 chunk 以 user_id doc_id chunk_id 作为检索与过滤核心信息

6 标签
   支持用户手工标签
   可选 AI 标签，策略是采样部分 chunk 让模型输出 2 到 3 个高质量标签
   标签只做文档级，不做 chunk 级

7 文档协同与分享
   当前方案采用文档级复制或 fork
   将一个 doc 复制到目标用户，使目标用户在 Milvus 与 MySQL 拥有独立的 doc_id 与 chunk
   这样避免跨用户的权限耦合，也避免在 Milvus 内做共享权限表


六 检索与 RAG 问答

需求对应

知识库构建与检索
语义检索
跨文档知识关联

RAGService 的典型链路

1 输入 query
   来自接口层或主对话 Agent
   支持可选 doc_ids 作为检索范围限制

2 embedding
   对 query 做向量化

3 混合检索
   在 Milvus 做 hybrid_search，召回规模偏大以保证覆盖
   同时利用稠密与稀疏信号，并用加权融合得到候选排序

4 rerank
   使用 rerank 模型对候选做精排
   典型策略是召回 top50 后 rerank 取 top10 或 top5

5 生成回答
   将重排后的 chunk content 拼接为上下文
   提示模型必须基于提供的片段回答
   输出回答文本与来源摘要

6 日志与可解释性
   将 query 与使用的 chunk 引用写入 RAGQueryLog 或等价日志表
   sources 返回给前端用于展示来源

跨文档关联如何实现

跨文档不是单独算法，而是让检索召回可以同时命中多个 doc_id
由于 Milvus 存的是所有文档 chunk，候选集中自然会包含多文档片段
最终 sources 返回 doc_id chunk_id 以及片段摘要即可


七 学习计划生成

需求对应

自动生成大纲
生成每日学习计划
支持动态调整

PlanService 与 PlanWorkflow

输入合同 PlanInput

user_id
goal_description
start_date end_date
daily_minutes
doc_ids 可选

工作流节点

1 expand_goal
   使用 LLM 将用户目标拆成若干主题 topics
   主题数量与学习天数约束相关，避免过多导致不可执行

2 retrieve_topics
   对每个主题调用 MilvusVectorStore.hybrid_search
   产出 topic_hits
   若某主题完全检索不到内容则放入 not_covered
   not_covered 用于提醒用户补充资料

3 build_syllabus
   将 topic_hits 的真实片段作为依据
   要求模型输出 JSON 结构的大纲
   大纲包含 unit 与 topic 层级，并给出 estimated_minutes
   如果总时长超出可用预算，会自动缩放或再次修正
   JSON 解析存在兜底策略，避免模型格式漂移导致中断

4 allocate_daily_plan
   将 syllabus 的分钟数按天拆分
   目标是每一天不超过 daily_minutes，最后一天兜底塞完
   如果某些天没有内容，会生成复盘或自测占位任务，保持计划连续

5 落库
   learning_plans 写入主记录
   daily_tasks 写入每日任务，包含 outline_json 与 status 初始为 pending

动态调整策略

当前方案属于覆盖式重算

当用户修改天数或每日分钟数或目标或资料范围
系统不做局部重排，而是将未完成任务标记取消或保留状态，再生成一份新 plan
这样代码简单，行为稳定，易于解释


八 测验生成与批改

需求对应

基于学习内容自动生成题目
支持即时答题与批改
记录答题情况识别薄弱环节

QuizService 与 QuizWorkflow

输入 QuizInput

user_id
description 出题意图
question_type 题型
doc_ids 可选

出题链路

1 召回
   将 description 做 embedding
   在 Milvus 进行混合检索召回 top50
   召回广以保证素材充足

2 重排
   使用 rerank 对 top50 重排取 top5
   top5 的完整 chunk 原文作为出题上下文

3 结构化出题
   通过 JSON Schema 约束题目结构
   题型覆盖 单选 多选 填空 简答

4 标准答案不回传
   generate 接口响应不返回 answer 字段
   标准答案存入 MySQL 的 quizzes.question_json
   这样前端无法直接看到答案，但后台可以在 submit 评分时读取

提交与评分链路

1 读取题目与标准答案

2 客观题判分
   单选 多选 填空通过规则对比
   当前实现偏保守，通常是完全匹配给满分，否则给低分或零分

3 主观题评分
   将题干 标准答案 用户答案交给 LLM
   输出 score 与 comment
   同样使用结构化 schema 并提供解析兜底

4 落库 attempt
   quiz_attempts 记录 answer_json score comment

薄弱点识别

当 score 低于阈值时生成 weak_points
weak_points 记录薄弱描述 关联 doc_id 与 chunk_id 列表
薄弱点将触发后续复习排期


九 复习提醒与学习复盘

需求对应

基于艾宾浩斯遗忘曲线
每日复习任务
学习与复习历史记录

ReviewService 设计

1 间隔表
   EB_INTERVALS 为 1 2 4 7 15 30 天
   属于离散间隔记忆法的工程近似

2 排期生成
   目标来源包括
   完成 daily_task
   产生 weak_point
   生成时保证幂等，同一 target_type target_id stage 不重复插入

3 今日查询
   list_today_reviews 按 scheduled_date 等于今天且 status 为 pending 查询

4 完成标记
   将 status 置为 completed 并写 completed_at
   这就是复习历史


十 主对话 Agent 与统一入口体验

需求对应

前后端可运行系统
将多个能力整合为统一对话体验

MainChatAgent 的职责边界

1 只做编排与路由
   不重复写业务逻辑
   RAG 计划 测验 复习各自有 Service 或子 Agent

2 意图识别
   规则优先，必要时调用 LLM 做结构化意图识别
   这样在外部服务不可用时仍能工作

3 参数透传
   user_id doc_ids db session 由主入口统一传递

子 Agent

PlanChatAgent
   负责从自然语言中抽取槽位并追问补齐
   槽齐全后调用 PlanService 生成计划

QuizChatAgent
   负责测验多轮状态
   例如先出题 再等待用户回答 再提交评分


十一 统计与可视化

需求对应

实现学习数据的可视化统计
学习时长 知识点掌握度等

StatsService 指标口径

1 total_learning_minutes
   近似口径为 完成任务天数乘以 plan.daily_minutes
   这是无埋点情况下的稳定估算

2 completed_task_days
   近 7 天游学任务完成过的天数

3 average_mastery_score
   基于 quiz_attempts 的平均分映射到 0 到 100

4 active_weak_points
   按 weak_points 未解决的数量统计

5 last_7_days_trend
   近 7 日趋势，输出每天 minutes 与状态 completed pending skipped none

前端呈现建议

当前仓库提供了简单前端页面示例
面试展示时可在页面渲染 KPI 卡片 趋势图 薄弱点列表
后端只提供稳定 JSON 输出


十二 协同编辑与分享

需求对应

支持学习资料的协同编辑和分享

当前工程落地方式

以文档共享或复制为主
实现策略

1 文档可 fork 给其他 user
   在 MySQL 为目标用户插入新的 documents 记录
   并将原文 raw_text 或解析缓存重建后重新分块入 Milvus

2 共享边界清晰
   不做跨用户权限表
   也不在 Milvus 中混入共享可见性字段

需求对应

支持语音输入和语音播报

十四 安全性 成本与稳定性设计

1 密钥管理
   API Key 通过环境变量注入
   config.py 只保存环境变量名，不保存密钥内容

2 embedding 缓存
   llm_client.py 使用 embedding_cache.json 落盘
   对重复文本可明显节省成本与时间

3 降级与容错
   langgraph 不存在时工作流顺序执行
   结构化解析失败时走兜底内容
   缓存读写失败不影响主流程

4 可观测性
   RAG 查询日志记录 query 与引用 chunk
   计划与测验关键步骤都有日志输出
